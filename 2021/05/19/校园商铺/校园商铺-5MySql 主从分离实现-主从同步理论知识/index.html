校园商铺-5MySql 主从分离实现-主从同步理论知识

1. 学习目标

  • 数据库层面的主从配置实现
  • 代码层面的读写分离实现 (无需改动现有代码)

大型网站为了减轻服务器处理海量的并发访问所产生的性能问题采用了很多的解决方案，其中最主流的方案之一便是读写分离，即将读操作和写操作分别导流到不同的服务器集群去执行，到了数据业务层、数据访问层，如果还是传统的单靠一台服务器去扛，面对如此多的数据库连接操作 数据库必然会崩溃，如若造成数据的丢失后果是不堪设想的。本章的目标就利用 MySql 的主从配置实现读写分离，以期达到减轻数据库服务器压力的目的。

2. 主从库理论知识

2.1 主从同步如何工作

主从同步的工作主要有三步：

  • 第一步：Master（主服务器）将对数据的操作记录到二进制日志 Binarylog即binlog 当中。也就是说在每个事物更新数据完成之前 Master 在日志里记录这些改变, MySql 将事物串行的写入二进制日志当中，在事件写入二进制日志完成之后，Master 通知存储引擎提交事务。提交好了事务就会进入到第二步。对数据的操作就称为是一次二进制的日志事件 Binarylog events。
  • 第二步：Slave将 Master 的Binarylog events拷贝到他的中继日志 Relay log 当中。首先，Slave 会开始一个工作线程，就是图中的 I/O thread ，I/O thread 在 Master 上打开一个普通的连接，然后开始做 binlog 的 Dumprocess ,就是将 binlog 拷贝到 I/O thread ，binlog dumprocess 从 Master 的 Binarylog 中读取事件，如果已经跟上 Master 的话，他就会睡眠并等待 Master 产生新的事件，I/O thread 将这些事件就写入到 Relay log 里面。到此第二部就完成了。
  • 第三步： 就是 Slave 重做中继日志事件。比如 Binarylog 对数据操作a、b、c，此时 Slave 还没有进行，就需要重做一遍，即将 Master 上的操作a、b、c在 Slave 重做一遍，做到数据当中， SQL thread 就是处理该过程的最后一步，SQL线程从中继日志 Relay log 读取事件，并重放其中的事件，更新Slave的数据 使其与Master中的数据一致。通过这三步咱们对Master操作就能反映到Slave上面了。
  • 工作流程总结：首先就是主服务器会将数据的变化记录到二进制日志当中,举个例子来说就是本来Master 是没有数据 a 的，此时我们这里添加了一条数据 a ，添加数据 a 的这个操作就会被记录到 Binarylog 当中，接下来从服务器 Slave 就会将这个添加 a 的操作同步到自己的 Relay log 当中.其次,再通过另外一个线程将这个同步的操作去执行，往从服务器里面插入一条 a 的数据这样两边的数据就能保持一致了。

为什么需要 Relay log 而不只是直接将 Binarylog 中记录的数据变化搬到从服务器中？

​ 从计算机的设计合理角度来说这些操作是需要队列去处理的，由于网络等原因 Binarylog 是不能一口气传送到 I/O thread 里来的。因此为了不影响从服务器执行别的操作，I/O thread 这里就会 new 一个I/O thread 出来，去接收主服务器传送过来的数据，并将这些数据存入到 Relay log 当中。Relay log 就类似于生产者和消费者里面的一个管道，就是用来缓存 Binarylog 传送过来的一些操作，其实 Relay log 也是存储在从服务器的缓存当中的，因此他的开销也是很小的。同时还 new 了一个消费者 SQL thread ,他也会异步的从 Relay log 里面去读取数据出来去消费，就是将这些操作实际的做进这个从服务器的数据当中，通过生产者消费者的模式，就达到不相互影响并且能够让 Slave 继续去做别的事情。
